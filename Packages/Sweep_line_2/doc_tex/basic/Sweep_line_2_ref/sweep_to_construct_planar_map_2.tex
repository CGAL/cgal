\ccDefinition
  
The function \ccRefName\ constructs a planar map from the set of 
interior disjoint subcurves induced by the given input curves.

\ccInclude{sweep_to_construct_planar_map_2.h}


\def\ccLongParamLayout{\ccTrue} 

\ccGlobalFunction{
   template <class InputIterator, class SweepLineTraits_2, class Planar_map>
   void 
   sweep_to_construct_planar_map(
   InputIterator curves_begin, 
   InputIterator curves_end, 
   SweepLineTraits_2& traits,
   Planar_map &result);}

constructs a \ccc{Planar map} from the curves given in the range
\ccStyle{[curves_begin, curves_end)}. The parameter \ccStyle{result}
holds the resulting \ccc{Planar map}. If \ccStyle{result} holds a
non empty \ccc {Planar map} then \ccStyle{result} will be
cleared and the curves \ccStyle{result} was holding would be sweeped
together with the curves in \ccStyle{[curves_begin, curves_end)}.

\ccc{sweep_to_construct_planar_map} supports curves which are
interior intersect, overlap or are non $x$-monotone. 
%and hence the user may provide his curves as input without any restriction of the kind
%mentioned above. 
This function constructs the \ccc{Planar map}
efficiently, since it makes no point location operations when
inserting curves to the \ccc{Planar map}. Hence, inserting curves to a
\ccc{Planar map}, even when there are no interior intersections
between those curves and only iteratively inserting them to a
\ccc{Planar map} is legal, will be more efficient when using
\ccc{sweep_to_construct_planar_map}.

\ccHeading{Requirements}
\begin{enumerate}
   \item    \ccc{InputIterator::value_type} is equivalent to \ccc{Traits::Curve_2}.
   \item    \ccc{SweepLineTraits_2} is a model of the \ccc{SweepLineTraits_2} concept
   \item    \ccc{Planar_map} is the \cgal Planar Map.
\end{enumerate}

% \ccImplementation 
% The implementation uses the container $map$ defined in STL for
% implementing the event queue and the status line.  The implementation
% also keeps per each input curve an associated container of all its
% intersection points ordered from left to right, and also keeps per
% each event point all its outcoming curves.  Due to these structures,
% when constructing a subdivision no point location operation has to be
% done. The complexity of this algorithm is $O(nlogn + k)$ where $n$ is
% the number of the input curves and $k$ is the number of intersection
% points induced by these curves.
